# AI-пайплайн

В проекте три независимых AI-задачи:

- найти человека в кадре;
- понять, кто именно попал в кадр;
- оценить вовлеченность.

Все три задачи живут отдельно и собираются только в `TrackAttendanceUseCase`.

## Детекция и трекинг людей

Файл: `src/backend/infrastructure/ai/person/detector.py`

Используется `YOLO` из `ultralytics` с `classes=[0]`, то есть только класс `person`. Для трекинга берется `ByteTrack`.

Что важно по реализации:

- модель загружается из `src/backend/assets/models/yolov8n.pt`;
- при наличии `CUDA` модель уходит на `cuda`;
- на выходе use case получает не просто bbox, а bbox вместе с `track_id`.

`track_id` здесь критичен. Через него потом стабилизируется имя ученика и сглаживается уровень вовлеченности.

## Распознавание лица

Файл: `src/backend/infrastructure/ai/face/recognizer.py`

### Основной режим

Приоритетный backend сейчас такой:

- `facenet-pytorch`
- `MTCNN` для поиска и выравнивания лица
- `InceptionResnetV1` для эмбеддинга
- `torch` на `CUDA`, если устройство доступно

Логика довольно простая:

1. При старте читаются все фото из `assets/images`.
2. Для каждого изображения считается эмбеддинг.
3. Эмбеддинги складываются в память.
4. На видео для нового лица считается текущий эмбеддинг.
5. Кандидаты группируются по `student_id`.
6. Побеждает тот, у кого лучшая distance и достаточный отрыв от второго кандидата.

### Запасной режим

Если `facenet-pytorch` не смог подняться, recognizer откатывается на `DeepFace`. Это нужно не для красоты, а чтобы не терять работоспособность из-за битого кэша весов, плохой сети или отсутствующего пакета.

Минус такого режима очевиден: в типичной Windows-сборке `DeepFace` часто остается на CPU.

### Почему имя не появляется мгновенно

Распознавание специально стабилизируется по `track_id`. Это сделано против ложных срабатываний.

У recognizer есть:

- окно голосования `FACE_VOTE_WINDOW`;
- минимальное число совпадений `FACE_MIN_STABLE_VOTES`.

Пока этот порог не набран, use case считает личность неподтвержденной.

## Оценка вовлеченности

Файл: `src/backend/infrastructure/ai/person/pose.py`

### Текущий порядок выбора backend

1. `yolo_pose`
2. `mediapipe_legacy`
3. `mediapipe_tasks`
4. `heuristic_face`

При наличии `yolov8n-pose.pt` проект старается взять именно `YOLO pose`, потому что это самый прямой путь на GPU.

### Почему после pose есть fallback

На живом потоке pose не обязан отработать на каждом кадре. Человек может сидеть боком, половина корпуса может выйти из кадра, а bbox может оказаться слишком тесным. Поэтому если pose не дал валидные keypoints, estimator откатывается к face-based варианту и все равно возвращает `low`, `medium` или `high`.

Иначе интерфейс быстро превращается в набор `unknown`.

### Что именно считает heuristic fallback

Fallback не пытается угадать эмоции. Он смотрит на более простые признаки:

- насколько лицо находится в ожидаемой зоне верхней части тела;
- насколько лицо смещено от центра;
- насколько крупно оно выглядит;
- насколько форма bbox похожа на фронтальный ракурс.

Это грубее, чем нормальная pose/landmark-модель, но для потока из аудитории работает терпимо и не требует дополнительных файлов.

## Как AI сшивается в один проход

Файл: `src/backend/use_case/track_attendance.py`

На каждом кадре происходит один и тот же цикл:

1. `PersonDetector` находит людей.
2. `FaceRecognizer.detect_faces` ищет лица на всем кадре.
3. Для каждого человека use case пытается сопоставить лицо с его head-region.
4. По лицу определяется `student_id`.
5. По человеку считается вовлеченность.
6. Когда человек держится в кадре достаточно долго, use case пишет attendance-log.

## Что влияет на точность сильнее всего

Не размер модели и не наличие GPU. На практике больше всего влияют:

- качество трех регистрационных фото;
- освещение в кабинете;
- высота камеры;
- насколько сильно ученики перекрывают друг друга;
- правильно ли подогнаны `FACE_DISTANCE_THRESHOLD` и `FACE_DISTANCE_MARGIN`.

GPU нужен в первую очередь для скорости и устойчивости на потоке. Сам по себе он точность не улучшает.
